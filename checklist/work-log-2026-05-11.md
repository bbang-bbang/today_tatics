## 4. 전술 카드 race condition 근본 해결

5/7 작업분 이후 사용자 보고: "특정 매치에서 전술 보기 카드가 보였다 안 보였다 한다". 3단계 시도 끝에 근본 원인 제거.

### 4-1. 1차 시도 — 매치 active 피드백 + 즉시 fetch (commit `44671a3`)
- 매치 카드 클릭 시 `.kmc-active` 클래스 토글 → 시각 피드백.
- 매치 전환 race: 직전 fetch 완료 시점에 다른 매치 선택 시 잘못된 렌더링 발생 → `_lastHome/_lastAway` 가드.

### 4-2. 2차 시도 — pending 보관 (commit `d539574`)
- match-extras 응답이 render() 완료 전 도착하면 `_pendingTactics`에 보관 → render 완료 후 `tryRenderPendingTactics()`로 재시도.

### 4-3. finished 무관 fetch (commit `0a08fcb`)
- 기존엔 `isFinished`일 때만 extras 시도 → 진행 중/예정 매치에서 데이터가 있어도 미표시. 항상 fetch 후 응답에 따라 표시.

### 4-4. 3차 (근본) — Promise.all 통합 (commit `41c6818`)
- 위 펜딩 흐름이 `workers=3` + `HTTP/2` 환경에서 여전히 일부 케이스 미표시.
- **조치**: match-extras를 `loadPrediction`의 `Promise.all`에 포함. render() 직후 동기적으로 `renderTacticsCard` 호출 → race 자체 제거.
- `_pendingTactics`, `tryRenderPendingTactics` 등 펜딩 흐름 전부 삭제. click handler 단순화.

### 4-5. cache-bust (commit `96abf75`)
- prediction.js / insights.js / style.css 버전 일괄 상향 (race fix 즉시 반영).

## 8. 핵심 발견 / 교훈

1. **race fix는 가드보다 통합** — pending 보관 + 가드 변수 + 재시도 루프로 막아도 멀티 worker + HTTP/2 환경에선 빈틈. 같은 시점에 함께 fetch해 `Promise.all`로 묶는 게 근본 해결.

2. **fail-closed 원칙** — `UPDATE_SECRET=None`일 때 검증 우회로 누구나 트리거 가능했던 패턴. 환경변수 미설정은 "허용"이 아니라 "거부(503)"로 처리.

3. **default_server 명시** — 같은 호스트의 다른 site config와 80 포트 공유 시 alphabetical 매칭에 의존하면 IP 접근/매칭 안 되는 호스트가 엉뚱한 site로 라우팅됨. 의도하는 site에 `default_server` 명시 필수.

4. **데이터 부족한 폴백은 UX 함정** — 예정 경기에서 H2H 데이터를 보여주는 게 친절해 보이지만, 사용자는 "이 경기 예측 데이터"로 오해. 데이터 없으면 그냥 미표시가 정직.

5. **인증 도입은 UI 정리와 묶기** — P7 인증 적용하면서 노출돼있던 ↻ 위젯이 403 받게 됨. 보안 보강은 항상 "이제 동작 안 하는 UI" 정리와 한 세트.

6. **HTTPS는 인증보다 먼저** — OAuth 3종 도입 전에 HTTPS + Secure 쿠키 + SameSite=Lax + HSTS를 먼저 확정. 평문에서 세션 도입하면 첫 트래픽이 위험.

7. **deploy를 묶지 않은 핫픽스는 핫픽스가 아니다** — 5/11 03시에 commit + push까지 하고 운영 pull 안 했음. 사용자는 8시간 동안 "보고서 안 나옴" 상태에 있었음. git push 자체로 끝났다고 착각. 한 줄로 묶는 deploy 스크립트 또는 commit-then-deploy 습관 필요.

8. **운영/로컬 DB 스키마는 같아도 row가 다르다** — 미래 매치 schema row(score=NULL)는 events 테이블에 있어야 정상(스케줄 표시용). 그런데 그것을 `ORDER BY date_ts DESC LIMIT N`로 가져가는 모든 비교 쿼리에서 None TypeError 잠복. 로컬엔 그 row가 없으니 안 터졌고 운영에서만 터짐. **events.home_score를 직접 비교하는 모든 쿼리는 NULL 필터 강제 코딩 컨벤션화 필요**. SUM/CASE 기반은 NULL safe.

9. **사용자 보고는 그대로 가장 빠른 진단** — "예측 보고서가 안 나옴 → 로컬 정상 → 운영 prediction.js v=27 확인 → 운영 main.py 옛 코드 → error log → NULL TypeError"의 진단 체인은 30분 안에 끝났음. 추측보다 사용자 환경(=운영)에서 직접 호출해보는 게 빠름.

10. **외부 API 좌표 시스템은 표면별로 다르다** — 같은 SofaScore API여도 avg_positions(자기 골=0)와 shotmap(공격 골=0)이 정반대 좌표계를 씀. 한 화면에서 두 데이터를 겹치려면 한쪽을 정규화해야 함. **저장 시점에 통일**(B안)이 가장 깔끔하나 이번엔 프론트 변환(A안)으로 즉시 fix. 추후 backlog의 백엔드 정규화 작업으로 이관 권장.

11. **lineup 기반 분석 SQL이 mps 기반보다 견고** — match_lineups는 모든 등록 선수(선발+벤치)를 항상 보유. match_player_stats는 매치별로 수집 누락 가능성. sub 분석처럼 "등록되었지만 mps 누락 가능"한 시나리오는 lineup을 출발점으로 LEFT JOIN하는 게 안전.

12. **MutationObserver는 의도치 않은 cross-module 통신 통로** — info.js가 사이드바 배너 텍스트 변경을 감지해 `teamsSelected`를 dispatch하는 게 일반 흐름엔 OK지만, app.js가 `applyMatchLineup`으로 textContent를 갱신할 때도 동일하게 trigger됨. 프로그램이 변경한 textContent와 사용자가 변경한 textContent를 구분 못함. 같은 정보(매치 ID)가 이미 알려진 상태(`_lastHome/_lastAway`)면 무시하는 idempotent 가드가 race 해결책.

13. **비결정적 race는 "한 매치는 잘 됨, 다른 매치는 안 됨"으로 보임** — race는 항상 똑같이 깨지지 않음. 수원-부산처럼 textContent가 그대로면 trigger 안 됨. 다른 매치는 trigger됨. 같은 매치도 새로고침 시 outcome 달라짐. 사용자 입장에서 "재현이 안 된다"는 보고가 race의 시그니처.

14. **외부 ID 잠복 — player.team_id는 master 데이터 아님** — 2군/유스/K3 등록 선수가 1군 매치에 임시 출장하면 SofaScore가 player.team_id를 외부 클럽 ID로 유지. 우리 자동화는 K1/K2 SS_ID 화이트리스트 기준이라 이 선수들은 모든 백필 phase에서 누락. 해결책: lineup의 실제 출장 매치(events.home/away_team_id)를 통한 역추적. **player 자체의 메타데이터에 의존하지 말고 매치에서의 컨텍스트를 신뢰**.

15. **데이터 부재와 데이터 잘못은 다른 표현이어야** — mps 미수집 매치에서 모든 starter mins=0을 "starter 11명 OUT"으로 분류하면 사용자 화면에 "교체 OUT 11개·IN 0개"라는 왜곡된 데이터가 표시됨. 실제로는 "데이터 없음"이 정확. 0과 NULL 또는 데이터 부재를 같이 처리하지 말 것. 부재 시그널이 분류 로직에 명시적으로 들어가야 함(`0 < mins < 90` 같은 양쪽 가드).

16. **QA는 전수 통계 + 패턴 분류가 단발 사용자 보고보다 빠름** — 사용자 "교체 리스트가 안 나옴" 보고에서 어떤 매치인지 묻기 전에, 166 match-side 전수 SQL로 카테고리 분포(OUT-only/IN-only/balanced) 뽑으니 21건 OUT-only 패턴이 즉시 드러남. 매치 하나 하나 추적보다 패턴 분류가 근본 원인까지 빠르게 도달.

17. **추정 데이터를 fix하려고 가드 쌓는 것보다 원본 데이터로 가는 게 깔끔** — mps의 minutes_played로 OUT/IN을 추정하는 로직에 가드를 계속 더하다가(mins=0 제외, 88→90 임계값 조정, 페어 매칭 정렬 등) 본질적 부정확성(같은 시각 동시 교체, 추가시간 출장)은 해소 불가. SofaScore가 `incidentType="substitution"` 형태로 정확한 OUT/IN/시각을 이미 주는데도 우리가 안 받고 있었음. 추정 로직에 N번째 가드를 추가하기 전에 "원본 데이터가 이미 있지 않나?" 자문할 것.

18. **사용자 의도는 반복 확인 — UI 토글 의미는 작은 단어 차이로 완전히 달라짐** — "선발/교체"가 "선발만 / 교체된 선수만"인지, "선발만 / 교체 포함"인지, "시작 라인업 / 종료 라인업"인지 한 마디로 결정되지 않음. 첫 추측은 거의 빗나가니까 첫 결과물 보여주고 사용자 confirm 받는 사이클이 가장 빠른 의도 합치 경로.

19. **백필 단계의 NULL/덮어쓰기 가드는 인풋 분포 바뀌면 다시 새는 곳이 생긴다** — 7-10 phase2가 "NULL or empty"만 덮어썼는데, 다른 phase가 영문을 미리 채워놓는 시나리오는 막지 못함. 가드는 "내가 안전하다고 판단한 상태만 통과"가 아니라 "최종으로 원하지 않는 상태(=영문)도 덮어쓴다"로 작성해야 함. 후자가 더 idempotent.

20. **단발 휴리스틱 핀인은 의외로 실용적** — 사후 분석 카드의 핀인 5종(결정력/세트피스/PK/자책골/조기 퇴장)이 추정 모델이나 통계 부담 없이 "사용자가 매치 보면서 가장 궁금한 한 줄"을 자동 생성. xG-실득 차이 0.8 같은 단순 임계값으로도 5/3 suwon_fc 케이스의 "결정력 폭발"이 정확히 잡힘. 큰 모델보다 도메인 지식 휴리스틱이 더 빠르게 가치를 줌.
